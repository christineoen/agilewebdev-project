AI Reflections

How I used AI tools throughout this project, what they got right, where they fell short, and what that means for working with them responsibly.

My Approach

I came into this project with nearly a decade of professional software engineering experience, which shaped how I chose to work with AI tools. Rather than delegating entire features and accepting the output wholesale, I treated each AI tool as a fast first draft — something to react to and correct rather than something to ship directly.

My general workflow throughout the project was:

  • Write requirements and rubric criteria into markdown files under a guidelines/ folder before touching any code
  • Give those guidelines to the AI as context for each task
  • Review the output carefully, make targeted corrections, and inspect the result in the browser before moving on
  • Commit and push manually from the terminal — never through the AI tool

This kept me in control of what went into the repository and meant I could catch problems before they compounded.

Use Case 1

Project Scaffolding with ChatGPT

Prompt

"Here is the full project specification for a web fundamentals assignment. How would you organise the folder structure? What files do I need to create?"

(Full project spec pasted as context.)

What the AI Produced

ChatGPT returned a sensible top-level folder structure: an index.html at the root, subfolders for css/, js/, data/, and images/, and placeholder HTML files for each page. It also suggested separating concerns by keeping JavaScript out of HTML files — advice that turned out to be more aspirational than the later AI tools managed in practice.

Changes I Made

I added a guidelines/ folder that wasn't in the AI's suggestion — a place to store the rubric and my own notes for each page so future AI prompts would have structured context to work from. This was my own addition and turned out to be one of the more useful decisions of the project.

Limitations Noticed

ChatGPT's suggestions were generic — it had no knowledge of the site's eventual colour palette, Bootstrap usage, or the specific rubric requirements for each page. The structure it produced was a reasonable starting point but not a finished plan. It also couldn't anticipate how the pages would evolve, so some of the empty files it suggested were later reorganised.

Use Case 2

Page Generation — Kiro vs Claude Code

Prompt (Kiro — Index page)

"Create the index page based on the requirements in guidelines/index.md. Use Bootstrap for layout and styling. Update the colour defaults to match these preferences: [colour values]."

What the AI Produced

Kiro handled the index page well. It applied Bootstrap correctly, picked up on the colour preferences, and produced clean, readable HTML. I accepted most of what it generated with only minor tweaks.

Prompt (Tutorial page — first attempt with Kiro)

"Create the tutorial page based on guidelines/tutorial.md. Follow the same structure and colour palette as the index page."

What the AI Produced

The tutorial page Kiro produced didn't look the way I wanted. The layout felt off and the interactive sections weren't structured how I had envisioned them. I scrapped it and gave the same prompt to Claude Code instead.

Claude Code's Output — and What I Had to Fix

Claude Code's version looked significantly better, but it had several structural problems:

  • Custom styles were written inline in <style> tags inside the HTML file rather than in styles.css
  • It introduced many bespoke CSS rules that duplicated or overrode Bootstrap utilities unnecessarily
  • All JavaScript was embedded directly in <script> tags at the bottom of the HTML rather than in a separate file

I followed up with corrective prompts:

"Use Bootstrap utility classes where you can instead of writing custom CSS. Move all custom styles into styles.css."

"Move all JavaScript into a separate file: js/tutorial.js."

After those corrections the structure matched the rest of the project. I also reviewed all of the interactive sections myself and made changes to improve usability — in particular the JavaScript demo, where I wanted the actual code being executed to be visible to the user as they interacted with it. I also updated the DOM manipulation section to use the <template> tag, which I was already familiar with and felt was a cleaner approach.

Limitations Noticed

The most consistent limitation across both Kiro and Claude Code was a tendency to solve problems in whatever way was immediately available rather than the way that was most consistent with the project's conventions. Neither tool had a strong sense of "this project already does X this way" without being told explicitly. They also both defaulted to inline code organisation unless specifically instructed otherwise — not a good default for maintainable projects.

Use Case 3

Code Explanation with GitHub Copilot

Prompt

"Explain what this section of code is doing." (Highlighted specific blocks in the tutorial page.)

What the AI Produced

GitHub Copilot gave clear, accurate plain-English explanations of the code Claude Code had generated — particularly useful for understanding IntersectionObserver usage in the tutorial scrolling sidebar.

Limitations Noticed

Copilot's explanations were generally accurate at the line level but occasionally missed the broader context — explaining what a function did without flagging whether it was the right function for the situation. That judgement still required me to bring my own knowledge of the problem.

Critical Evaluation

Reliability

AI tools were reliable for well-scoped, clearly described tasks — generating a component from a spec, explaining a code block, or suggesting a folder structure. They became less reliable when the task required awareness of project-wide conventions or involved trade-offs that weren't spelled out in the prompt. The quality of output was directly proportional to the quality of context provided.

Risks

The most significant risk I observed was over-acceptance — the temptation to ship AI output without reading it carefully because it looks correct. Generated code is syntactically plausible and often runs without errors, but that doesn't mean it does what you intended, follows the project's conventions, or is maintainable. For a developer without prior experience, this risk would be much higher.

A secondary risk is organisational: without the guidelines/ structure I put in place, it would have been easy for different AI tools to pull the project in different directions, producing inconsistent patterns across pages.

Ethical Considerations

Using AI to assist with an assessed assignment raises legitimate questions about authorship. My position is that the judgement, structure, corrections, and final decisions were mine — the AI acted as a fast typist for a first draft, not as the author of the work. But I think it's important to be honest about that distinction rather than obscuring it, which is why this page exists and why AI assistance is credited explicitly in the references section of the CV page.

More broadly, AI tools trained on public code carry licensing and attribution questions that aren't fully resolved. For a commercial project I would be more cautious about accepting generated code without understanding its provenance.
